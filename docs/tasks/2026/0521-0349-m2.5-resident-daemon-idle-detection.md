# feat: M2.5 resident daemon with built-in idle detection (GNOME Mutter IdleMonitor)

## Overview

Today howan is a single-shot process: `howan start` calls `app::run()`
(`crates/howan/src/app.rs:83`), which opens the saver window, blocks in a
calloop event loop until the first input (or `SIGTERM`/`SIGINT` from
`howan stop`), then exits. The original activation design delegated idle
detection to an external `swayidle` watcher that would invoke
`howan start` / `howan stop` (see the module docs in
`crates/howan/src/main.rs:1` and `crates/howan/src/cli.rs:1`).

That split no longer works on the primary target. **Mutter does not implement
`ext-idle-notify-v1`** — it only offers `zwp_idle_inhibit_manager_v1` (idle
*inhibit*, not *detection*) — so `swayidle` fails to start on GNOME with
`Compositor doesn't support idle protocol`. The only justification for the
external-watcher split — reusing an off-the-shelf idle tool — is gone, and idle
detection now has to live inside howan either way. So howan becomes a **single
resident daemon** that owns idle detection, surface display, and (in later
milestones) the phased lifecycle.

This task implements the daemon core and GNOME idle detection only:

1. **Resident daemon mode.** Add a `howan daemon` subcommand that runs a
   long-lived process. It connects to Wayland but does **not** show a surface
   immediately. It subscribes to an idle source. When the idle threshold `T1`
   (default 5 minutes, overridable on the command line — see criterion 6)
   fires, the daemon shows the saver surface. On the first `wl_seat` input it
   destroys that surface and **stays resident**, re-arming for the next idle.
   This is the key behavioral change from M1/M2: input destroys the *surface*,
   not the *process*. `SIGTERM`/`SIGINT` still terminate the whole daemon
   cleanly (the systemd stop path).

2. **`IdleSource` abstraction with a single GNOME backend.** Define an
   `IdleSource` trait (or equivalent seam) that yields "idle reached `T1`"
   events to the daemon loop, and implement exactly one backend against it:
   GNOME's `org.gnome.Mutter.IdleMonitor` D-Bus interface via `zbus`. The trait
   exists so a future `ext-idle-notify-v1` (wlroots) backend can be added later
   **without changing the daemon loop** — but no second backend is implemented
   in this task, and none is required (other compositors are nice-to-have, not a
   target). The daemon loop must depend on the trait, not on the concrete Mutter
   type.

3. **Decouple surface lifecycle from process lifecycle.** `HowanApp` currently
   always holds a `Window` (`crates/howan/src/app.rs:199`). Refactor so the
   saver surface is created on demand when idle fires and dropped on dismiss,
   while the durable Wayland state (connection, registry, seat, output,
   compositor/xdg/shm globals) persists across show/hide cycles. Showing then
   hiding then showing again within one process must work repeatably.

The saver surface keeps the **composited-surface invariants** verified in PR #3:
no `set_fullscreen`, no opaque region (see the long safety comment at
`crates/howan/src/app.rs:127` and `docs/guides/30-composited-surface.md`). The
daemon must not regress those — recreating the surface each idle cycle must
recreate it the same safe way.

The manual/debug `start` / `stop` CLI is **kept as-is** (immediate show / SIGTERM
to a one-shot run); it is no longer the activation path but stays for manual
testing.

  input, and DPMS delegation in Phase 3 (M4). This task tracks the idle-start

- Multi-monitor coverage of all outputs (M5) — the daemon targets the active

- GPU / shader rendering (M6+); the renderer stays on `wl_shm`.

  plus a single CLI override flag is the only configuration in scope.

## Implementation notes

- **Bridging async zbus into the sync calloop loop.** `zbus` is async, while the
  Wayland side runs on calloop (single-threaded, see
  `crates/howan/src/app.rs:93`). The recommended approach is to run the D-Bus
  idle watch on a dedicated thread (zbus's blocking API or a small async
  runtime) and forward "idle reached `T1`" events into the calloop loop via a
  `calloop::channel::channel()` source, keeping all Wayland work on the main
  thread. Integrating the zbus connection fd directly into calloop is an
  alternative but is fiddlier; prefer the channel bridge unless there is a clear
  reason not to. Whatever the choice, the daemon loop should observe idle events
  through the `IdleSource` trait, not the transport details.
- **Re-arming for subsequent cycles.** Verify how Mutter's `AddIdleWatch`
  behaves after the user becomes active again — whether the watch re-fires on
  the next idle period or must be re-added (possibly paired with
  `AddUserActiveWatch` / `GetIdletime`). Pick a mechanism that reliably produces
  an idle event on every subsequent idle period and document it in the guide.
- **Surface recreation.** Splitting the always-present `Window`
  (`crates/howan/src/app.rs:199`) into a recreatable saver surface is the bulk of
  the refactor. Keep the durable Wayland state (registry / seat / output /
  compositor / xdg / shm) alive across cycles; recreate only the window +
  renderer buffer per show. The seat input handlers in
  `crates/howan/src/app/handlers.rs` dismiss on input — adapt them so dismiss
  tears down the surface and signals the daemon loop rather than setting a
  process-exit flag. Note that today both the input handlers **and** the
  `SIGTERM`/`SIGINT` signal source (`app.rs:108`) funnel through the same
  `dismiss()` / `exit` flag; in the daemon these two paths must diverge — input means
  "tear down surface, re-arm, stay resident", while `SIGTERM`/`SIGINT` must
  still terminate the whole daemon (criterion 5). Do not let the dismiss
  refactor neuter the signal path into a surface-only teardown.
- **`start`/`stop` and the PID file.** Keep `howan start` / `howan stop` working
  as today (one-shot show / SIGTERM via `crates/howan/src/pidfile.rs`). Make sure
  the daemon does not clobber or get unintentionally killed by the one-shot
  PID-file semantics; if the daemon participates in the PID file at all, document
  the chosen behavior. Note the concrete collision: `PidFileGuard::acquire()`
  rejects launch when a live instance already owns the file, and `stop()`
  SIGTERMs whatever PID it finds — so a daemon sharing that file would make
  `howan start` error and `howan stop` kill the daemon. Decide deliberately
  whether the daemon writes the PID file, uses a distinct one, or none. Keep this
  coherent and minimal — do not redesign the IPC.
